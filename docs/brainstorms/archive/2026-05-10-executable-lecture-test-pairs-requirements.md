# Executable Lecture-Test Pairs

## Summary

Executable lecture pairs for all 14 lecture docs (00–13): each lecture gets a standalone Python script that runs a config, computes the analytical reference from the derivation, generates an overlay plot, and asserts physics matches within tolerances. When physics diverges, the code is fixed before moving on. Existing regression configs and `verify_*.py` scripts are refactored into lecture-pair format where they overlap. Make targets provide the entry point.

---

## Problem Frame

The codebase has 14 lecture docs with detailed derivations, 20+ regression configs with golden-output comparison, and 4 standalone verification scripts (`verify_qwz_chern.py`, `verify_bhz_z2.py`, `verify_landau_levels.py`, `sweep_rashba_bdg.py`). These are disconnected: reading a lecture doc doesn't verify the code, running a regression test doesn't reference the physics derivation, and the verification scripts aren't tied to any lecture.

The gap is a single workflow that connects derivation to code output. When a new module is added or the Hamiltonian construction changes, there's no systematic way to check which physics derivations are still valid. The existing verification scripts prove the pattern works for topological modules but cover only a fraction of the 14 lectures — the exact mapping of each script to lectures is deferred to planning.

This work turns each lecture into an active validation instrument. The fix-as-you-go approach means every lecture pair produces a verified result: either the physics matches the derivation, or the code has been corrected to make it match.

---

## Key Flows

- F1. **Build and validate a lecture pair**
  - **Trigger:** Developer starts work on a lecture (e.g., `make lecture-05`)
  - **Actors:** Developer
  - **Steps:** (1) Script writes or reuses a config; (2) runs the relevant executable; (3) parses output; (4) computes the analytical reference from the lecture derivation; (5) generates overlay plot comparing code vs. analytical; (6) asserts physics within tolerance; (7) if divergence, investigates and fixes code, then re-runs.
  - **Outcome:** Lecture pair passes — code output matches analytical derivation. Overlay plot is committed and embedded in lecture doc.
  - **Covered by:** R1, R2, R3, R5, R9

- F2. **Refactor existing verification script into lecture pair**
  - **Trigger:** An existing `verify_*.py` script covers physics from a lecture doc
  - **Actors:** Developer
  - **Steps:** (1) Identify which lecture the verification script maps to; (2) restructure script to follow the lecture-pair pattern (config, analytical reference, overlay plot, assertion); (3) add Make target; (4) embed generated figure in lecture doc.
  - **Outcome:** Existing verification script becomes a first-class lecture pair with the same interface as new ones.
  - **Covered by:** R7, R4, R5

## Requirements

**Script infrastructure**

- R1. Each lecture (00-quickstart through 13-topological-superconductivity) gets a standalone Python script that: writes or identifies a config, runs the relevant executable, parses output, computes the analytical reference, generates an overlay plot, and asserts physics within documented tolerances.
- R2. Existing regression configs from `tests/regression/configs/` are reused when they exercise the relevant physics. New configs are added for coverage gaps and follow the existing naming convention.
- R4. The Makefile provides a `make lecture-XX` target for each lecture and a `make lectures` target that runs all pairs sequentially.
- R11. Overlay plots generated by scripts are committed to version control under the lecture figures directory.

**Validation workflow**

- R3. When code output diverges from the analytical derivation, the code is investigated and fixed before advancing to the next lecture. A pair is not done until physics matches.
- R9. Tolerances are lecture-specific and documented inline in each script. Known model limitations (e.g., the 20-30% g-factor shortfall vs. experiment in the 8-band model) are flagged as documented limitations, not treated as test failures.
- R10. Scripts produce rich diagnostic output to support the investigation workflow: overlay plots with both code and analytical curves, and numerical comparison tables showing expected vs. computed values.

**Lecture doc updates**

- R5. Overlay plots generated by lecture scripts are embedded in the corresponding lecture doc, replacing or supplementing existing static figures where applicable.
- R6. Lecture doc derivations are updated with explicit code-output anchors — statements like "running config X produces Y eV" that connect the mathematical derivation to the code result.
- R8. Lectures without clean analytical comparisons (00-quickstart, 12-extending-the-code) get worked-example configs that demonstrate the concepts with runnable code, even without numerical assertions against published values.

**Refactoring existing infrastructure**

- R7. The existing 4 standalone verification scripts in `scripts/` (`verify_qwz_chern.py`, `verify_bhz_z2.py`, `verify_landau_levels.py`, `sweep_rashba_bdg.py`) are refactored into lecture-pair format where they cover the same physics as a lecture doc. The refactored scripts follow the same pattern as new lecture scripts.

## Acceptance Examples

- AE1. **Covers R1, R3.** Given lecture 05 (g-factor), when the script runs `gfactorCalculation` with the GaAs bulk config and computes the Roth formula analytical value, the code output matches within tolerance (e.g., GaAs CB g ≈ -0.315). If the 8-band model produces a different value due to a code bug, the bug is fixed and the script re-run until it passes.
- AE2. **Covers R7, R4.** Given the existing `verify_qwz_chern.py` script (which validates Chern number computation against literature), when it is refactored into a lecture pair for lecture 13 (topological superconductivity, section 13.2), it gains a `make lecture-13` target and its output plot is embedded in the lecture doc. Note: `verify_landau_levels.py` targets the `bandStructure` executable and maps to Landau level physics (potentially lecture 05 or lecture 13 section 13.4.3), not the topological invariant workflow — the script-to-lecture mapping must be verified during planning.
- AE3. **Covers R8.** Given lecture 00 (quickstart), which has no analytical comparison, the script runs a simple bulk GaAs config and generates a band-structure plot demonstrating the key concepts, without numerical assertions.
- AE4. **Covers R9.** Given lecture 05 (g-factor), when the 8-band model produces a g-factor that diverges from experiment by the known 20-30% model limitation, the script documents this as a known limitation rather than failing.

## Success Criteria

- Running `make lectures` produces 14 passing pairs with overlay plots embedded in all lecture docs.
- Each lecture doc has at least one code-output anchor connecting a derivation equation to a computed value.
- Existing `verify_*.py` scripts are absorbed into the lecture-pair system (no orphaned verification scripts remain).
- A downstream planner can take any single lecture pair and understand exactly what physics is being validated, what the analytical reference is, and what tolerance is acceptable.

## Scope Boundaries

- Jupyter notebook integration (adds dependency burden; can follow if the standalone scripts prove insufficient for learners)
- Automated CI integration of lecture pairs (scripts run manually via `make lecture-XX` for now)
- Creating a shared lecture framework or generic runner (YAGNI — factor out shared utilities later if duplication becomes painful across 14 scripts)
- Adding new physics modules or lectures beyond the existing 14
- Changing the regression test infrastructure or `compare_output.py` comparison mechanism

---

## Key Decisions

- **Standalone scripts over shared framework (Approach A):** The fix-as-you-go workflow requires heavy per-script debugging. Standalone scripts are the easiest to debug, extend, and iterate on. Shared utilities can be extracted later if duplication hurts.
- **Fix-as-you-go over report-then-fix:** Each pair is done when physics matches. This produces a verified codebase at every step rather than accumulating a backlog of discrepancies.
- **Reuse + extend configs:** No duplication — existing configs serve double duty where they exercise the right physics. New configs are only added for genuine gaps.
- **All 14 lectures in first pass:** Including methodological ones. Even lectures without analytical comparisons get worked-example configs, ensuring complete coverage.
- **Refactor existing verify_*.py scripts:** The 4 existing verification scripts are proto-lecture-pairs. Absorbing them gives immediate coverage for the topological lectures and eliminates orphaned infrastructure.

## Dependencies / Assumptions

- The 14 lecture docs contain derivations with sufficient detail to extract analytical reference values. Lectures that lack this will need derivation updates (in scope per R6).
- The Fortran executables produce parseable output, but the four executables write different file formats: `bandStructure` writes `eigenvalues.dat`, `gfactorCalculation` writes `gfactor.dat`, `opticalProperties` writes `absorption_TE/TM.dat`, and `topologicalAnalysis` writes `topology_result.dat`. Each lecture script will need a per-executable parser. Existing parsers in `tests/integration/verify_*.py` scripts (e.g., `parse_eigenvalues` in `verify_landau_analytical.py`) can be reused as reference.
- The existing `verify_landau_levels.py` in `scripts/` has a bug: it reads from `band_results.dat` (which does not exist) instead of `eigenvalues.dat`. This must be fixed before refactoring it into a lecture pair.
- The existing `generate_all_figures.py` infrastructure may need updating as new lecture scripts are added.
- Python 3 with matplotlib and numpy is available (already a dependency for regression test comparison).
